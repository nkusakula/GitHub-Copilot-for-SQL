# GitHub Copilot — Database Instructions & Custom Instructions in SSMS
### Customer Demo Script | AdventureWorks Database

---

## Setup Checklist (Before the Demo)

1. **SSMS 22.3+** installed with the AI Assistance workload
2. Signed into GitHub account with Copilot access
3. Connected to the `adventure_works` database
4. Copilot Chat window open: **View → GitHub Copilot Chat**
5. A text editor ready to create the `copilot-instructions.md` file

---

## Overview — What Are We Showing?

| Feature | Where It Lives | Who Sets It Up | Who Benefits |
|---|---|---|---|
| **Database instructions** | Inside the database (extended properties) | DBA / data owner | Every user of that database |
| **Database constitution** | Inside the database (extended properties) | DBA / data owner | Every user of that database |
| **Custom instructions** | A markdown file on disk | Individual developer | That developer, in all sessions |

> *"Think of it this way: database instructions teach Copilot about **your data**. Custom instructions teach Copilot about **you** — how you like to work and write code."*

---

## Part 1 — Database Instructions

### What Are Database Instructions?

> *"Database instructions are stored directly in the database as extended properties. They travel with the database — any user who connects with Copilot automatically gets the benefit of them. You set them once, and Copilot applies them forever."*

---

### Demo 1 — Expose the Problem: Ambiguous Business Logic

**Talking point:**
> *"Let me show you a situation where Copilot gives a technically correct answer but a business-incorrect one — because it doesn't know your rules yet."*

**Step 1:** In the Copilot Chat window, start a new thread (`Ctrl+N`) and ask:

```
What is the total revenue generated by each product category in AdventureWorks?
```

**What to observe:** Copilot will likely sum `LineTotal` from `SalesOrderDetail` joined to `ProductCategory`. This is a reasonable guess — but note that it has no idea whether your business defines revenue as `LineTotal`, `UnitPrice * OrderQty`, or something else. It also doesn't know whether cancelled or returned orders should be excluded.

**What to say:**
> *"Copilot gave us a plausible query — but in a real business, 'revenue' often has a specific definition. Maybe we exclude certain order statuses. Without telling Copilot what we mean, every developer might write this differently."*

---

### Demo 2 — Add a Database Instruction via Copilot

**Talking point:**
> *"Now I'm going to teach Copilot what 'revenue' means for this database — and it will remember it for everyone."*

**Step 1:** In the same or a new chat thread, type:

```
Add a database instruction for the SalesLT.SalesOrderDetail table to state that
revenue is defined as SUM(LineTotal) only for orders where the parent
SalesOrderHeader.Status = 5 (shipped orders). Cancelled or in-progress orders
should be excluded from revenue calculations.
```

**Step 2:** Copilot generates the T-SQL to save this as an extended property. It will look like this:

```sql
-- Review carefully before executing
EXECUTE sp_addextendedproperty
    @name  = N'AGENTS.md',
    @value = N'Revenue is defined as SUM(LineTotal) only for orders where the
               parent SalesOrderHeader.Status = 5 (shipped). Exclude orders
               with Status <> 5 from all revenue calculations.',
    @level0type = N'SCHEMA',
    @level0name = N'SalesLT',
    @level1type = N'TABLE',
    @level1name = N'SalesOrderDetail';
```

**Step 3:** Copy the generated T-SQL, paste it into a new query window, review it, and **execute it**.

**What to say:**
> *"Notice Copilot can't run this itself — it generates the script and I execute it. This is by design. Copilot operates read-only in Ask mode; data modifications always go through you."*

### Demo 3 — Add a Second Instruction for a Column

**Talking point:**
> *"Instructions aren't just for tables. You can add them to specific columns too. Let's annotate a column that has a non-obvious encoding."*

**Step 1:** In the chat, type:

```
Add a database instruction for the Status column in SalesLT.SalesOrderHeader
to explain that Status is encoded as: 1=In Process, 2=Approved, 3=Backordered,
4=Rejected, 5=Shipped, 6=Cancelled.
```

**Step 2:** Copilot generates:

```sql
-- Review carefully before executing
EXECUTE sp_addextendedproperty
    @name  = N'AGENTS.md',
    @value = N'Status column encoding: 1=In Process, 2=Approved, 3=Backordered,
               4=Rejected, 5=Shipped, 6=Cancelled.',
    @level0type = N'SCHEMA',
    @level0name = N'SalesLT',
    @level1type = N'TABLE',
    @level1name = N'SalesOrderHeader',
    @level2type = N'COLUMN',
    @level2name = N'Status';
```

**Step 3:** Execute the script.

**What to say:**
> *"Now every developer who asks Copilot about order status will get the right answer without having to look up a data dictionary or ask a colleague."*

### Demo 6 — Database Constitution

**Talking point:**
> *"Beyond per-object instructions, you can set a single database-wide constitution — the highest precedence rule that applies to every Copilot interaction on this database."*

**Step 1:** In the chat, type:

```
Create a database constitution for adventure_works that states: all queries must
be schema-qualified, SELECT * is not permitted, and all queries that could scan
large tables should include a TOP clause or a date filter.
```

**Step 2:** Copilot generates:

```sql
-- Review carefully before executing
EXECUTE sp_addextendedproperty
    @name  = N'CONSTITUTION.md',
    @value = N'All T-SQL generated for this database must comply with these standards:
1. Always schema-qualify object names (e.g., SalesLT.Customer, not just Customer).
2. Never use SELECT * - always specify column names explicitly.
3. Queries that could scan large tables must include a TOP clause or a restrictive
   date/key filter to prevent unintended full table scans.';
```

**Step 3:** Execute the script.

**Step 4:** Test it immediately — in a new thread, ask:

```
Write a query to get all customers.
```

**What to observe:** Copilot will qualify with `SalesLT.Customer`, name columns explicitly, and likely add a filter or `TOP` — all driven by the constitution.

**What to say:**
> *"The constitution is the guardrail. It applies to every user, every session, without exception. Set it once at the database level and your whole team writes consistent, safe T-SQL."*

### Managing Instructions (Quick Reference)

```sql
-- Update an existing instruction
EXECUTE sp_updateextendedproperty
    @name       = N'AGENTS.md',
    @value      = N'Updated instruction text here.',
    @level0type = N'SCHEMA',
    @level0name = N'SalesLT',
    @level1type = N'TABLE',
    @level1name = N'SalesOrderDetail';

-- Remove an instruction
EXECUTE sp_dropextendedproperty
    @name       = N'AGENTS.md',
    @level0type = N'SCHEMA',
    @level0name = N'SalesLT',
    @level1type = N'TABLE',
    @level1name = N'SalesOrderDetail';
```

## Part 2 — Custom Instructions

### What Are Custom Instructions?

> *"While database instructions are about the data, custom instructions are about the developer. They let each person tell Copilot: 'Here's how I like to work.' Copilot silently applies them to every response — no one on the team sees them, but every response reflects them."*

There are **two levels**:

| Level | File Location | Scope |
|---|---|---|
| **Repository-level** | `.github/copilot-instructions.md` in repo root | Applies when working in that repo |
| **User-level** | `%USERPROFILE%\copilot-instructions.md` | Applies across all sessions |

---

### Demo 7 — Create a User-Level Custom Instructions File

**Talking point:**
> *"I'll create a personal instructions file that lives in my user profile. From this point on, every Copilot response in SSMS will follow my preferences — coding style, safety rules, naming conventions — automatically."*

**Step 1:** Open Notepad (or any text editor) and create a file at:

```
C:\Users\<YourUsername>\copilot-instructions.md
```

**Step 2:** Paste in this content:

```markdown
# Copilot Instructions

## Persona
- I am an experienced SQL Server developer working with production AdventureWorks data.
- Use a concise, professional tone. Explain your reasoning briefly.
- When multiple approaches exist, mention the tradeoffs.

## Safety and Execution Guardrails
- Treat this as a production database unless I say otherwise.
- For queries that could scan large tables, always add a TOP 100 or date range
  filter first, and offer the full version separately.
- Never generate DROP, TRUNCATE, or DELETE without an explicit WHERE clause.

## Code Style
- Always schema-qualify object names (e.g., SalesLT.Customer, not Customer).
- Use INNER JOIN / LEFT JOIN syntax, never implicit joins with commas.
- Place each JOIN condition on its own line with ON on a new line.
- Always alias tables with short, readable abbreviations (c for Customer,
  soh for SalesOrderHeader, sod for SalesOrderDetail, p for Product).
- Use UPPERCASE for all T-SQL keywords (SELECT, FROM, WHERE, GROUP BY).
- Never use SELECT * — always list columns explicitly.

## Naming Conventions
- Tables: PascalCase singular (Customer, Product)
- Stored procedures: usp_<Action><Entity> (e.g., usp_GetCustomerOrders)
- Views: v_<Description> (e.g., v_CustomerRevenueSummary)
- Functions: fn_<Description> (e.g., fn_GetOrderTotal)
- Temp tables: #<PascalCase> (e.g., #TopCustomers)

## Documentation
- Add inline comments for any non-obvious JOIN condition or filter.
- For stored procedures and views, include a header comment block with:
  Author, Date, Description, Parameters, and Change History.
```

**Step 3:** Save the file.

**Step 4:** Verify the feature is enabled in SSMS:
**Tools → Options → GitHub → Copilot → Copilot Chat** → check **Enable custom instructions**.

### Demo 8 — See Custom Instructions in Action

**Talking point:**
> *"The instructions are invisible in the chat window — Copilot silently applies them. Watch the difference in code style."*

**Step 1:** Start a new thread (`Ctrl+N`) and ask:

```
Write a query to show the top customers by revenue with their order count and
average order value.
```

**What to observe in the response:**
- Table aliases match our conventions (`c`, `soh`)
- Keywords are uppercased (`SELECT`, `FROM`, `INNER JOIN`)
- Schema-qualified (`SalesLT.Customer`, not just `Customer`)
- No `SELECT *`
- A `TOP` clause or filter applied (safety guardrail)
- JOIN conditions on separate lines

**What to say:**
> *"I didn't ask for any of that in my prompt. It's all coming from the custom instructions file. Every query Copilot writes for me will follow these standards automatically."*

**Step 2:** Check the **References** section in Copilot's response — it will list `copilot-instructions.md`, confirming the file was applied.

## Comparison: Database Instructions vs. Custom Instructions

| | Database Instructions | Custom Instructions |
|---|---|---|
| **Stored in** | Database (extended properties) | Markdown file on disk |
| **Set by** | DBA / database owner | Individual developer |
| **Applies to** | All users of the database | The individual developer only |
| **Teaches Copilot about** | Business rules, column meanings, domain logic | Coding style, safety rules, naming conventions |
| **Persists across machines** | Yes (travels with the database) | Only if the file is shared/synced |
| **Visible in chat** | No (applied silently) | No (listed in References only) |
| **Format** | `AGENTS.md` extended property | `copilot-instructions.md` markdown file |

---

## Closing Summary

**Use database instructions when:**
- A column or table has non-obvious meaning (e.g., Status codes)
- Revenue, KPIs, or business metrics have a specific definition
- You want every developer on the team to get consistent Copilot answers
- You want living documentation that travels with the database

**Use custom instructions when:**
- You want Copilot to always follow your personal coding style
- You want consistent naming conventions across everything you generate
- You want safety guardrails applied automatically (no SELECT *, no large scans)
- You want stored procedures to always include header documentation

**Closing line:**
> *"Together, database instructions and custom instructions transform Copilot from a generic AI assistant into a domain expert that knows your data and speaks your team's language — without you having to explain it every time."*
